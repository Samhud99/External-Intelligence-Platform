# External Intelligence Platform - Proof of Concept Design

**Date:** 2026-02-27
**Status:** Approved

---

## Goal

Build a minimal proof of concept that demonstrates the core EIP loop: a user describes what external information they want to monitor in natural language, an AI agent discovers the source and generates an extraction strategy, and an automated runner executes that strategy on a schedule — storing structured results as JSON.

This validates the PRD's two-phase model: agent discovers once (expensive), automation runs cheaply.

---

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Scope | Proof of concept | Validate core loop before building full platform |
| Stack | Python + FastAPI | Strong web scraping ecosystem, fast to prototype |
| AI model | Pluggable (Claude first) | Abstract provider interface, start with Claude, swap later |
| Storage | File-based JSON | Zero infrastructure for PoC, easy to migrate later |
| Output | JSON API / file output | No external integrations needed for PoC |
| Approach | Agent-first pipeline | Two-phase model validates the core PRD thesis |

### Core Flow

1. User sends natural language request to `POST /jobs/create`
2. Setup agent receives request, fetches target URL, analyzes page structure
3. Agent determines CSS selectors / extraction strategy, validates by running a test extraction
4. Agent iterates if results aren't satisfactory
5. On success: saves job definition + extraction config, returns sample results
6. User confirms, job goes live on schedule
7. Automated runner executes extraction config on each scheduled run
8. Runner compares results against previous run to flag new/changed items
9. If extraction fails (site changed), job marked `needs_reagent` for re-discovery

## Agent Layer

### Model Provider Interface

```python
class ModelProvider(Protocol):
    async def complete(self, system: str, messages: list[dict], tools: list[dict] | None = None) -> dict: ...
```

Concrete implementations: `ClaudeProvider` (default), with room for `AzureOpenAIProvider`.

### Agent Tools

The setup agent has access to:

- `fetch_page(url)` — fetch a URL and return HTML content
- `extract_with_selectors(url, selectors)` — test CSS selectors against a page, return extracted items
- `save_job(job_definition, extraction_config)` — persist the job and config

### Setup Agent Flow

1. Receives natural language request from user
2. Calls `fetch_page` to examine the target site
3. Reasons about page structure, identifies selectors
4. Calls `extract_with_selectors` to validate strategy
5. Iterates if results aren't good
6. Calls `save_job` to persist final config
7. Returns sample results + job summary

### Re-agenting

When the automated runner hits extraction failures (empty results, parse errors), it marks the job as `needs_reagent`. The same agent flow runs again to rebuild the extraction config.

## API Endpoints

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/jobs/create` | Natural language job creation via setup agent |
| `GET` | `/jobs` | List all jobs |
| `GET` | `/jobs/{id}` | Get job details + extraction config |
| `POST` | `/jobs/{id}/run` | Manually trigger a run |
| `PATCH` | `/jobs/{id}` | Update job (pause, resume, change schedule) |
| `DELETE` | `/jobs/{id}` | Delete a job |
| `GET` | `/jobs/{id}/results` | Get results for a job |
| `GET` | `/jobs/{id}/results/{run_id}` | Get a specific run's results |

---

## Automated Runner

- Uses `httpx` to fetch pages and `BeautifulSoup` to apply CSS selectors from extraction config
- Compares extracted items against previous run's results (by URL or title hash) to flag new items
- On failure: retry once, then mark job as `needs_reagent`, log the error
- HTTP errors: exponential backoff, max 3 retries, then fail the run

---

## Scheduler

- `APScheduler` with cron triggers parsed from each job's `schedule` field
- Runs in-process as a background task in the FastAPI app
- On startup, loads all active jobs and schedules them

## Error Handling

- Agent failures (model errors, tool errors): return error to user, don't save job
- Runner extraction failures: retry once, then mark `needs_reagent`, log error
- HTTP errors (rate limiting, 404s): exponential backoff, max 3 retries, then fail run
- No global error swallowing — all errors logged and surfaced

---

## Testing Strategy

- Unit tests for extraction runner, change detection, and store layer
- Integration test: create a job against a known stable page, verify extraction works
- Mock the model provider for agent tests
